1. Architecture Blueprint & decoupled Topology

SysOptima Sentinel is a high-fidelity, hybrid Endpoint Detection and Response (EDR) platform designed for Windows enterprise clients. Traditional signature-based detection systems fail to identify fileless attacks, dynamic LSASS credential dumping, and advanced ransomware execution trees. SysOptima addresses these gaps by evaluating cross-process behavior, parent-child lineages, temporal access patterns, and signature trust heuristics.

The platform's architecture is strictly decoupled into two core logical spaces, separating high-privilege telemetry capturing from behavioral analysis and user orchestration:

  1. Elevated C++ EDR Sensor (Kernel Telemetry Agent): Runs in elevated context as a native compiled binary. It hooks kernel trace providers without custom kernel drivers, formatting and packaging telemetry before pumping it over a secured Named Pipe IPC boundary.
  2. User-Space Python Cortex (Correlating & Operations Engine): Consumes telemetry, performs real-time sliding-window deduplication, maintains the NetworkX process behavior graph, validates certificate chains, makes graduated mitigation decisions, and serves the glassmorphic tactical operator dashboard.

2. ETW Telemetry Capture (KrabsETW Engine)

Event Tracing for Windows (ETW) provides low-overhead, high-performance tracing hooks built directly into the Windows NT kernel. By leveraging ETW, the SysOptima C++ Sensor achieves kernel-level visibility without executing dangerous kernel drivers (which are complex to digitally sign with WHQL, and present high crash-risks like BugChecks).

Using Microsoft's open-source KrabsETW library, we declare filters for process spawn trees, file system operations, and registry configurations.

Structured Telemetry Packets

Once captured by ETW callbacks, events are serialized into custom structured binary packets. The EDR data structure matches this exact memory layout:

Struct Field Data Type Byte Offset Description
event_type uint32_t Offset 0 Numeric event code (e.g. 1 = Process Start, 2 = File Write).
process_id uint32_t Offset 4 PID of the process triggering the event.
parent_id uint32_t Offset 8 Parent PID (populated for process creation events).
timestamp uint64_t Offset 12 Win32 FILETIME timestamp (100-nanosecond intervals since 1601).
payload_len uint32_t Offset 20 Size of the variable-length UTF-8 payload (file path or process name).
payload char[] Offset 24+ Dynamic string payload (variable length).

C++ KrabsETW Process Hook Callback Implementation

// Surgical C++ process creation intercept callback
void SetupProcessTrace(krabs::kernel_trace& trace) {
    krabs::kernel::process_provider process_prov;
    
    process_prov.add_on_event_callback([](const EVENT_RECORD& record) {
        krabs::schema schema(record);
        
        // ETW Kernel Opcode 1 represents Process Creation
        if (schema.opcode() == 1) {
            uint32_t pid = schema.any_property<uint32_t>(L"ProcessId");
            uint32_t parentPid = schema.any_property<uint32_t>(L"ParentId");
            std::wstring imagePath = schema.any_property<std::wstring>(L"ImageFileName");
            
            // Package into struct
            TelemetryPacket packet;
            packet.event_type = 1; // EVT_PROCESS_START
            packet.process_id = pid;
            packet.parent_id = parentPid;
            packet.timestamp = record.EventHeader.TimeStamp.QuadPart;
            
            // Convert wide characters to UTF-8 for transport
            std::string pathUtf8 = ConvertWideToUtf8(imagePath);
            packet.payload_len = pathUtf8.length();
            
            // Dispatch to secure named pipe IPC queue
            PipeQueue::Push(packet, pathUtf8);
        }
    });
    
    trace.enable(process_prov);
}

3. Secure IPC Boundaries (DACL & SDDL Deep Dive)

Because the C++ Sensor executes inside a privileged security context (NT AUTHORITY\SYSTEM) and hooks global syscall tracing, it acts as a critical system asset. If an unprivileged user process could write into `\\.\pipe\SysOptimaControl` or read raw logs from `\\.\pipe\SysOptimaData`, it could bypass EDR detection or inject fake whitelists.

To resolve this security boundary risk, SysOptima Sentinel locks IPC access down strictly using a hardcoded **Security Descriptor Definition Language (SDDL)** DACL.

Decoding EDR Security Descriptor Token Strings

The SDDL configuration used to initialize named pipes is L"D:(A;;GA;;;SY)(A;;GA;;;BA)". Let's break this token string down:

  • D:: Declares the start of the Discretionary Access Control List (DACL).
  • (A;;GA;;;SY):
    • A: Access Allowed.
    • GA: Generic All Access (full read, write, create, control).
    • SY: Local SYSTEM Account (Kernel service engine).
  • (A;;GA;;;BA):
    • A: Access Allowed.
    • GA: Generic All Access.
    • BA: Builtin Administrators Group (Local administrators).

Any process running as a standard non-elevated user trying to bind or listen to this pipe receives a standard Win32 ERROR_ACCESS_DENIED (5) error, blocking unauthorized access.

4. Real-time Ingestion & Deduplication Window

High-frequency process workloads (e.g. compiling a C++ project or downloading large database updates) can trigger thousands of file writes and network sockets within a few seconds. If the EDR processed every single event through its cryptographic and behavioral graph engines, CPU usage would spike to 100%, slowing down the operating system.

To solve this performance bottleneck, the Python Cortex engine uses a thread-safe Deduplication Sliding Window.

By hashing the event payload (e.g. Target Path) and comparing timestamps within the window, subsequent high-frequency events are dropped. When a process issues 500 writes to the same log file in 200 milliseconds, only the first write event enters the behavior graph, dropping the duplicate load.

5. Temporal Graph Engine (Parent-Child Tracking)

The Behavior Graph Engine forms the intellectual heart of Cortex. Utilizing a memory-optimized directional Graph (networkx.DiGraph), it records processes and their interactions as nodes and edges.

Each node is uniquely tracked by its PID and initialized with complete system properties. Edges represent spawn relationships or I/O events.

Graph Schema Specifications

Nodes represent entities, categorized by distinct types:

Node Type Attributes Stored Behavioral Role
process pid, label, full_path, threat_level, trust_score, is_signed, origin, creation_time Represents executing binaries on the host system.
file path, write_count, last_modified Represents targeting files written, modified, or quarantined.
network destination_ip, port, socket_handle Represents established outbound sockets.

6. Certificate Trust & Authenticode Caching

Calculating program signature trust is key to eliminating false positives. Legitimate Windows system utilities or elevated security processes (like taskmgr.exe or DISM.exe) perform intensive directory indexing and registry adjustments. An EDR without trust verification would flag these as malicious (e.g. heuristic ransomware or registry tampering alerts).

SysOptima Sentinel features a Genuine Windows Authenticode Trust Engine. When a process launches, Cortex queries its digital signature.

🚀 Performance Caching

Executing signature checks on disk binaries using native Windows DLLs (like wintrust.dll or running PowerShell queries) takes approximately 250ms to 400ms. If executed synchronously on every single thread spawn, the host system would lock up.

SysOptima resolves this by running signature checks asynchronously and caching the results in a thread-safe dictionary cache. Once a path's trust score is calculated, subsequent executions are fetched in less than 0.05 milliseconds.

Authenticode Verification & Cache Logic

import subprocess
import threading

class AuthenticodeTrustEngine:
    def __init__(self):
        self._cache = {}
        self._lock = threading.Lock()
        
    def get_trust_verdict(self, file_path: str) -> dict:
        # Thread-safe dictionary cache lookup
        with self._lock:
            if file_path in self._cache:
                return self._cache[file_path]
        
        # Fallback: Query Windows Authenticode Signature via powershell
        verdict = self._verify_on_disk_signature(file_path)
        
        with self._lock:
            self._cache[file_path] = verdict
        return verdict
        
    def _verify_on_disk_signature(self, path: str) -> dict:
        try:
            # Safe PowerShell digital signer lookup
            cmd = [
                "powershell.exe", "-NoProfile", "-Command",
                f"(Get-AuthenticodeSignature '{path}').SignerCertificate.Subject"
            ]
            res = subprocess.run(cmd, stdout=subprocess.PIPE, stderr=subprocess.PIPE, text=True, timeout=3)
            subject = res.stdout.strip()
            
            if res.returncode == 0 and subject:
                is_signed = True
                # Microsoft systems check
                trust_score = 100 if "Microsoft" in subject else 60
                origin = subject
            else:
                is_signed = False
                trust_score = 0
                origin = "Unsigned Binary"
        except Exception:
            is_signed = False
            trust_score = 0
            origin = "Verification Failed"
            
        return {
            "is_signed": is_signed,
            "trust_score": trust_score,
            "origin": origin
        }

7. Tactical Mitigations & Active Containment

When a process's threat score crosses the critical threshold (balancing verified threat behavior against dynamic trust states), the Response Orchestrator executes graduated tactical actions:

1. Smart Bottom-Up Tree Snipping

Traditional EDR engines often terminate the parent process first. However, advanced malware frequently spawns child threads or watchdog scripts that monitor the parent. If the parent dies, the child instantly detects it and spawns a new instance of the parent, rendering the block useless.

SysOptima Sentinel utilizes a **Smart Bottom-Up Snipping** algorithm. It traces the process tree lineage to find the youngest child leaves, sniping them first before moving up the process tree.

2. Outbound Network Firewall Isolation

When a process exhibits C&C beacon behavior, the EDR injects temporary firewall outbound blocks. By calling netsh advfirewall on the process's absolute on-disk path, all socket requests are blocked while keeping the process loaded in memory for security operations profiling: 

# Windows Defender Firewall PID network isolation block rule
netsh advfirewall firewall add rule name="SysOptima_EDR_Isolate_<pid>" dir=out action=block program="<filepath>" enable=yes

3. Process Memory Minidumper

To capture in-memory keys, unpacked payloads, or decrypters, the console features a live Win32 process memory minidumper. Calling MiniDumpWriteDump from dbghelp.dll, the dumper generates a complete memory image without needing external debugger engines: 

def dump_process_memory_win32(pid: int, output_path: str) -> bool:
    import ctypes
    
    # Win32 APIs constants
    PROCESS_ALL_ACCESS = 0x001F0FFF
    MiniDumpWithFullMemory = 2
    
    kernel32 = ctypes.windll.kernel32
    dbghelp = ctypes.windll.dbghelp
    
    # 1. Open target process handle
    h_process = kernel32.OpenProcess(PROCESS_ALL_ACCESS, False, pid)
    if not h_process:
        return False
        
    # 2. Open destination file handle
    GENERIC_WRITE = 0x40000000
    CREATE_ALWAYS = 2
    FILE_ATTRIBUTE_NORMAL = 0x80
    
    h_file = kernel32.CreateFileW(
        ctypes.c_wchar_p(output_path),
        GENERIC_WRITE, 0, None,
        CREATE_ALWAYS, FILE_ATTRIBUTE_NORMAL, None
    )
    
    if h_file == -1:
        kernel32.CloseHandle(h_process)
        return False
        
    # 3. Write native minidump
    success = dbghelp.MiniDumpWriteDump(
        h_process, pid, h_file, MiniDumpWithFullMemory, None, None, None
    )
    
    kernel32.CloseHandle(h_file)
    kernel32.CloseHandle(h_process)
    return bool(success)

8. Operations Dashboard UI Engine

The EDR console runs on a single-page **Vanilla CSS & HTML5** interface designed with a modern glassmorphic look. We use **Socket.IO** (WebSockets) to ensure that incoming events captured from Kernel ETW render on the UI in less than 50 milliseconds.

Operators can click on any node to query forensic details (Timeline, Registry, Files, Network Sockets) or click tactical overlays to trigger memory dumps or network isolation, directly sending command events down the named pipe back into the C++ process agent.

3. SQLite Telemetry Database Schema Layout

All ingested system events are stored inside the local SQLite database. Below is the structural schema definition: 

-- Core EDR Forensic Events Database Schema
CREATE TABLE IF NOT EXISTS security_events (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp REAL NOT NULL,
    event_type INTEGER NOT NULL,
    pid INTEGER NOT NULL,
    parent_pid INTEGER,
    process_name TEXT NOT NULL,
    executable_path TEXT,
    payload TEXT,
    threat_level INTEGER DEFAULT 0,
    trust_score INTEGER DEFAULT 0,
    verdict TEXT
);

CREATE INDEX IF NOT EXISTS idx_events_pid ON security_events(pid);
CREATE INDEX IF NOT EXISTS idx_events_timestamp ON security_events(timestamp);

4. Useful Forensic SQLite Queries

To search the EDR database for suspicious files written to sensitive system directories (like AppData or System32): 

-- Query written files in temporary or AppData user locations
SELECT datetime(timestamp, 'unixepoch') AS event_time, pid, process_name, payload AS target_file
FROM security_events
WHERE event_type = 2 -- EVT_FILE_WRITE
  AND (payload LIKE '%AppData%' OR payload LIKE '%Temp%' OR payload LIKE '%System32%')
ORDER BY timestamp DESC;
