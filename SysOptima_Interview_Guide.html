SysOptima EDR Architecture

The Ultimate Interview Guide: Complete breakdown of the 4-Brain paradigm, design decisions, libraries, and real-time behavioral malware detection engine.

Executive Summary

SysOptima is an enterprise-grade Endpoint Detection and Response (EDR) system positioned to rival commercial products like CrowdStrike or SentinelOne. It shifts from traditional signature-based detection to a dynamic, semantic behavioral analysis combined with unsupervised AI.

Our core philosophy relies on the 4-Brain Paradigm, creating a layered defense where high-volume event filtering occurs in high-speed native languages (C/C++), and high-complexity semantic intelligence occurs in highly capable reasoning environments (Python).

🧠 The 4-Brain Architecture

Brain 0: The Subconscious (Kernel Driver)

Language: C (WDK)

Purpose: Deep visibility into kernel-level techniques that user-mode APIs miss.

  • Hooks into PsSetCreateProcessNotifyRoutineEx and NtAllocateVirtualMemory.
  • Detects fileless malware, Process Hollowing, APC Thread Injections, and Direct Syscalls.
  • Why it matters: Modern malware bypasses user-mode hooks via direct syscalls. Our Ring-0 driver catches them dead in their tracks.

Brain 1: The Instinct (C++ Sentinel)

Language: C++17

Purpose: Microsecond reflex-based filtering and instant containment.

  • Consumes ETW (Event Tracing for Windows) at ~100k events/sec.
  • Aggregates noise (e.g. 50 file writes = 1 summary package).
  • Issues Instant Kill for known bad signatures or blatant spoofing (e.g., svchost.exe running from Downloads).
  • Maintains Event Reorder Buffer to output chronological events.

Brain 2: The Logic (Python Cortex)

Language: Python 3.11

Purpose: Semantic Pattern Recognition and Provenance Graphing.

  • Translates abstract OS events to Semantic Tags (e.g., Registry write to Run key = TAG_PERSISTENCE).
  • Tracks multiple sliding windows (Micro/Short/Long) to map MITRE ATT&CK patterns.
  • Executes atomic tree termination to prevent watchdog respawning.

Brain 3: The Intuition (AI Observer)

Language: Python (scikit-learn)

Purpose: Unsupervised machine learning to detect zero-day anomalies.

  • Collects baseline behaviors of normal usage to learn what is "safe".
  • Uses an ensemble model including Isolation Forest for outlier process detection.
  • Monitors Graph Drift (Graph Differencing) to immediately flag novel attacks.

⚡ Critical Engineering Decisions & Problems Solved

Decision 1: Combating the Python Bottleneck

The Trap: Python cannot process 100,000 raw events/sec, especially if we iterate through regex strings.

The Solution: The C++ Micro-Aggregator. If a process rapidly writes 100 files, C++ compresses it into {type: MASS_FILE_WRITE, count: 100}. For Python, we switched from raw string regex to pre-compiled regex, exact match caches, and Bloom filters, boosting processing from 100 to 10,000 translations/sec.

IPC: We bridge C++ and Python via 3 Binary Named Pipes (Data Stream, Control Stream, Query Stream) to prevent blocking over TCP/IP stacks.

Decision 2: Atomic Process Tree Termination

The Trap: Advanced malware spawns a "watchdog" process. If the EDR kills the malware, the watchdog instantly respawns it. If the EDR kills the watchdog, the malware respawns the watchdog.

The Solution: Graph-based Process Group Manager. First, we send a SUSPEND command to all processes recursively in the tree. Once the whole tree is frozen, we issue KILL commands in reverse order (leaves first). The malware literally cannot fight back.

Decision 3: Preventing AI Poisoning

The Trap: If malware runs during the initial learning phase, the AI will register the malware as "normal behavior".

The Solution: The Dual-Window Baseline Manager. The user initiates an explicit Training Mode session (e.g., 2 hours). During this time, the AI only learns; it does not block. The user ensures a clean environment. After the window closes, it locks the baseline and switches to Detection Mode.

Decision 4: Eliminating False Positives with Trust & Sandbox

The Trap: Memory scanning for RWX (Read-Write-Execute) regions often flags legitimate Software like Chrome, .NET runtimes, or Java because of Just-In-Time (JIT) compilation.

The Solution: We implemented the Memory Scanner with Trust-Based Whitelisting which cross-references known signatures and JIT heuristics before flagging. Furthermore, borderline suspicious apps (Score 40-60) are sent to the Sandbox Controller. The C++ engine forks the app into an isolated, net-less VM natively, observing it for 30 seconds to catch file-encrypting (Ransomware) or beaconing behavior.

Decision 5: Advanced Threat Containment (Quarantine)

The Trap: Blankly deleting detected threats ruins digital forensics.

The Solution: Implemented a secure Quarantine Manager. It hashes the threat, tracks metadata, isolates it structurally from execution privileges, and allows the sysadmin to restore it via the Dash Dashboard if it's deemed a false positive.

🔄 Lifecycle of an Attack: How SysOptima Responds

Scenario: Zero-Day Ransomware Attempt

  1. T+0s (Execution): User downloads and runs malware.exe. Driver detects ProcessCreate (unsigned). Python logs it. Suspicious Tag Applied.
  2. T+2s (Memory Hook): Malware attempts to map RWX memory. Driver flags it. Sentinel sends memory tags to Python. Score hits 100 (Critical). Python issues SUSPEND.
  3. T+2.5s (Sandbox Replay): Sentinel forks the suspended process into a sandbox. In 30 seconds, it observes file encryption behaviors. Pattern Matcher: RANSOMWARE_EXECUTION (100% confidence).
  4. T+3s (Atomic Kill & Cleanup): Sentinal executes KILL_TREE. Python issues QUARANTINE_FILE and triggers Persistence Cleanup to purge it from Startup/Registry. Dashboard alerts User.

Total Time to Neutralization: ~3 seconds. Zero damage passed the sandbox.
